# Лог ИИ-запросов (История промптов в процессе разработки MVP «Шеф-стол»)

В данном документе сохранена полная хронология общения с искусственным интеллектом в процессе выполнения технического задания по проектированию и разработке клиентского прототипа кулинарной студии «Шеф-стол». Лог отражает весь реальный путь: от черновых абстрактных вопросов до точечных багфиксов UI-стейта и логики отмены бронирований.

---

## 📋 Часть 1: Выявление бизнес-требований и ограничений (Этап 1)

1. **Запрос:** Привет! Мне дали учебное задание спроектировать клиентское приложение для кулинарной студии «Шеф-стол». Заказчик Артём жалуется на путаницу в WhatsApp и овербукинг. С чего начать сбор требований? Напиши верхнеуровневый план.
2. **Запрос:** Твой план слишком масштабный для MVP. У меня жесткие сроки — нужно сделать прототип всего за одну неделю. Помоги сузить требования, выкинув всё лишнее, чтобы успеть вовремя.
3. **Запрос:** Сгенерируй User Stories для этого приложения. Роли: Клиент студии, Администратор расписания, Шеф-повар.
4. **Запрос:** Стоп, тут изменение скоупа. Заказчик Артём уточнил, что админка для менеджеров и дашборд для шеф-поваров уже существуют в легаси-системе студии. Нам нужно спроектировать *только* интерфейс Клиента. Удали из требований истории админа и шефа.
5. **Запрос:** Я хочу, чтобы клиент мог просматривать интерактивное расписание и листать календарь мастер-классов на полгода вперед. Напиши под это функциональное требование.
6. **Запрос:** Переделай предыдущий пункт. Заказчик говорит, что продукты под мастер-классы закупаются строго под даты, поэтому расписание жестко ограничено — показываем классы строго на ближайшие 7 дней от текущей даты. Исправь это ограничение.
7. **Запрос:** Добавь в требования доп. услугу: платный прокат премиального инвентаря (именной фартук и японские ножи) за фиксированные 500 рублей. Как это лучше описать для пользователя?
8. **Запрос:** Нужно зафиксировать логику расчета стоимости: если клиент оставляет чекбокс пустым, цена базовая. Если выбирает прокат инвентаря — система должна интерактивно прибавлять 500 рублей к стоимости в окне записи. Запиши это в финансовые правила.
9. **Запрос:** Что делать, если студия отменяет класс из-за форс-мажора (например, шеф-повар заболел или сорвалась поставка редких ингредиентов)? Как приложение должно реагировать на статус отмены?
10. **Запрос:** Сформулируй итоговый набор User Stories по шаблону «Как клиент... Я хочу... Чтобы...» с учетом всех ограничений (7 дней, лимит 10 мест в студии, прокат инвентаря, отмена классов студией и 5-звездочная оценка шефа после занятия).
11. **Запрос:** Оформи получившиеся истории в красивую структурированную таблицу на Markdown для отчета по первому этапу (`1_REQUIREMENTS.md`).
12. **Запрос:** Теперь детально распиши основной пользовательский сценарий (User Flow): «Успешная запись клиента на мастер-класс по Итальянской пасте с выбором платной аренды фартука и ножей».
13. **Запрос:** Распиши альтернативный сценарий: «Отображение в ленте класса, отмененного студией». Карточка должна приглушаться, выводить причину отмены от Артёма, а кнопка записи — намертво блокироваться.
14. **Запрос:** Проверь получившийся файл требований на наличие логических противоречий. Всё ли сходится с бизнес-логикой кулинарной студии?

## 🏗️ Часть 2: Проектирование архитектуры и Mock-данных (Этап 2)

15. **Запрос:** Бизнес-логика готова. Переходим к архитектуре. Напиши серверный код на Node.js и схемы таблиц базы данных PostgreSQL для хранения расписания кулинарной студии.
16. **Запрос:** Отмена, я невнимательно прочитал задание! Никакого бэкенда и баз данных делать нельзя. Мы пишем чистое Single Page Application (SPA), а бэкенд для нас — это "black-box". Данные должны храниться прямо в оперативной памяти приложения в виде Mock-объектов. Спроектируй JSON-структуры для фронтенда.
17. **Запрос:** Опиши модель данных для кулинарного мастер-класса (`CookingClass`). Какие обязательные поля должны быть в JSON, чтобы поддержать логику мест, цен и статусов отмены?
18. **Запрос:** Добавь в JSON `CookingClass` поля: id, title, chef_id, datetime, duration, price (базовая цена), total_places (всего мест), available_places (остаток мест), статус (`available`, `full`, `cancelled`), cancel_reason (причина отмены) и rental_equipment_price.
19. **Запрос:** Опиши вторую JSON-модель — сущность Шеф-повара (`Chef`). Нам нужны поля: id, name и средний дробный рейтинг шефа от 1.0 до 5.0 для вывода на карточке.
20. **Запрос:** Спроектируй третью JSON-модель — Бронирование (`Booking`). В ней нужно хранить id брони, id класса, флаг выбора аренды инвентаря `with_rental` (true/false), итоговую стоимость `total_price` и поле для оценки шефу `chef_rating`.
21. **Запрос:** Напиши пример готового JSON-массива mock-данных на 7 дней вперед, содержащий минимум 4 мастер-класса (один доступен, один заполнен до отказа, один отменен студией) и двух разных шеф-поваров.
22. **Запрос:** Раздели наше клиентское SPA-приложение на архитектурные слои. Опиши обязанности Интерфейса (UI Layer), Бизнес-логики (Business Logic Layer) и Слоя данных (Mock Data Layer). Как они общаются?
23. **Запрос:** Собери все схемы данных, JSON-модели и описание слоев в единый архитектурный файл `2_ARCHITECTURE.md`.

## 💻 Часть 3: Разработка интерфейса, HTML и CSS (Этап 3)

24. **Запрос:** Переходим к коду. Сгенерируй мне структуру из пяти разных HTML-страниц для сайта: главная, страница класса, страница аренды фартуков, личный кабинет и страница отзывов.
25. **Запрос:** Нет, стой, это же должно быть Single Page Application (SPA). Нам нужен ровно один файл `index.html`. Внутри него сделай переключатель вкладок (Tabs) через кнопки: «Расписание классов» и «Мои бронирования». Под ними будут отображаться соответствующие контейнеры.
26. **Запрос:** Напиши полную валидную разметку `index.html` для мобильного экрана кулинарного MVP. Включи туда шапку студии, навигацию по двум вкладкам, пустой грид для карточек, скрытое модальное окно бронирования с чекбоксом проката и форму оценки шефа.
27. **Запрос:** Напиши стили `style.css` для этого HTML. Заказчик хочет уютную кулинарную тему: «кухонный тикет» — теплая бумажная подложка, мягкие тени, скругления карточек по 14-16px, красивые шрифты (сочетание встроенных или Inter/Fraunces) и верстка строго в один столбец под мобильный формат.
28. **Запрос:** Добавь в CSS стилизацию состояний карточек. Если класс отменен (`cancelled`), карточка должна визуально выцветать, а кнопка «Записаться» должна становиться серой, блеклой и полностью игнорировать клики пользователя.
29. **Запрос:** Опиши CSS-стили для центрирования модального окна бронирования. Оно должно плавно появляться поверх интерфейса, а задний фон (overlay) должен затемняться и слегка размывать расписание классов.
30. **Запрос:** Сделай интерактивные звездочки для формы оценки шеф-повара во вкладке «Мои бронирования». Напиши CSS-классы `.is-hover` и `.is-active`, чтобы они красиво меняли цвет при наведении и фиксации оценки.
31. **Запрос:** Обнаружена проблема в верстке: длинные названия мастер-классов (например, «Азиатский вок: техника работы с огнем и специи») ломают границы карточки. Напиши CSS-правило для аккуратного переноса или ограничения длинного текста.

## ⚙️ Часть 4: Реализация бизнес-логики на JavaScript (Этап 3)

32. **Запрос:** Напиши JavaScript-код для файла `script.js`. Начни с инициализации глобального состояния (State): добавь туда наши mock-данные классов с расчетом дат на 7 дней от `new Date()` и пустой массив `bookings` для бронирований.
33. **Запрос:** Создай функцию `renderClasses()`, которая перебирает массив стейта и динамически генерирует HTML-код карточек. Напиши ветвление: если свободных мест 0 — выводи плашку «Мест нет», если класс отменен — пиши причину отмены Артёма.
34. **Запрос:** Напиши обработчик клика по кнопке «Записаться». При клике функция должна находить нужный класс по ID, сохранять его в переменную активного выбора и открывать модальное окно бронирования, заполняя его поля.
35. **Запрос:** Добавь событие `change` на чекбокс «Аренда фартука и ножей (+500 руб)». Напиши логику, которая при клике на чекбокс мгновенно пересчитывает итоговую сумму в DOM: `базовая цена класса + 500`, если галочка стоит.
36. **Запрос:** Реализуй функцию подтверждения бронирования при отправке формы. Она должна собирать данные (имя, телефон клиента, флаг аренды, итоговую цену), генерировать объект брони и пушить его в массив `bookings`.
37. **Запрос:** Баг! При нажатии на кнопку «Подтвердить» в модальном окне страница мгновенно перезагружается, а добавленное бронирование стирается из памяти JS. Как это исправить? (Подсказка: отмени дефолтное поведение формы).
38. **Запрос:** Добавь строгую валидацию перед сохранением бронирования: если в объекте класса поле `availableSeats` равно 0, прерви выполнение функции бронирования и заблокируй отправку, чтобы избежать овербукинга.
39. **Запрос:** Напиши обновление стейта при успешной записи: модалка должна скрываться, поле `availableSeats` выбранного класса уменьшается ровно на 1, после чего автоматически вызывается перерисовка расписания `renderClasses()`, чтобы остаток мест обновился на экране без перезагрузки.
40. **Запрос:** Напиши функцию `renderBookings()` для отрисовки вкладки «Мои бронирования». Она должна генерировать список активных записей клиента с указанием темы, цены, флага аренды инвентаря и кнопкой «Отменить запись».
41. **Запрос:** Добавь в карточку бронирования функционал завершения класса: кнопку «Оценить шефа», которая открывает модальное окно со звездочками (1–5). При клике на звезду оценка должна записываться в поле `rating` объекта бронирования, а на экране выводиться сообщение «Спасибо за оценку!».

## 🐛 Часть 5: Отладка, исправление логических ошибок и сборка (Этап 4)

42. **Запрос:** Я провожу тестирование разработанного MVP и обнаружил критическую логическую ошибку в управлении состоянием: когда я записываюсь на класс, место списывается (было 3, стало 2). Но когда я перехожу во вкладку «Мои бронирования» и нажимаю «Отменить запись», бронь исчезает, а место в расписание обратно не возвращается! Как это исправить в коде?
43. **Запрос:** Напиши исправление для функции отмены: при удалении брони найди в массиве классов исходный мастер-класс по `class_id`, сделай инкремент (`availableSeats++`). Дополнительно: если у класса до этого был статус `full` (мест нет), переключи его обратно в `available` и сними блокировку с кнопки записи.
44. **Запрос:** Баг верстки: если на телефоне включена системная темная тема, цвет шрифта на карточках автоматически инвертируется браузером и сливается с фоном бумажной подложки. Исправь переменные в `:root` CSS, жестко закрепив темные тона для текста светлой темы.
45. **Запрос:** Для облегчения отладки добавь детальное логирование через `console.log()` во все ключевые функции `script.js`: логируй открытие модалки, клик по чекбоксу фартуков, создание структуры брони и моменты инкремента/декремента свободных мест.
46. **Запрос:** Проведи рефакторинг JavaScript-кода. Разбей его на понятные логические секции с помощью аккуратных комментариев (Вспомогательные функции дат, Mock-данные, Управление модальными окнами, Рендеринг UI, Обработчики событий), чтобы проверяющие эксперты легко оценили структуру проекта.
47. **Запрос:** Помоги написать итоговый файл `README.md` для моего GitHub-репозитория. Опиши концепцию MVP кулинарной студии «Шеф-стол», структуру получившихся файлов прототипа и добавь простую инструкцию для проверяющих, как запустить приложение (просто открыть `index.html` локально в любом браузере).
