# Нефункциональные требования

> Этап 2. Требования к качеству, архитектуре и ограничениям системы.

| ID | Категория | Описание требования | Приоритет |
| :-- | :-- | :-- | :-- |
| NFR-1 | Интерфейс | Адаптивная верстка интерфейса (HTML5/CSS3) для корректного отображения как на мобильных устройствах, так и на десктопах. | Высокий |
| NFR-2 | Производительность | Мгновенный отклик UI (< 100 мс) при фильтрации, переключении экранов и бронировании благодаря локальной работе с Mock Data Layer. | Высокий |
| NFR-4 | Архитектура | Реализация приложения строго в формате Single Page Application (SPA) на чистом JavaScript (Vanilla JS) без использования сторонних фреймворков. | Must |
| NFR-8 | Сохранение состояния | Сохранение всех изменений данных (новые бронирования, изменение счетчиков мест) в памяти приложения в рамках текущей сессии браузера. | Must |
| NFR-12 | Конфиденциальность | Клиент имеет доступ на просмотр и отмену только своих собственных бронирований, привязанных к его сессии. | Высокий |
| NFR-16 | Технические ограничения | Полное отсутствие внешних HTTP-запросов к серверу бэкенда; вся эмуляция работы с данными инкапсулирована в локальном JS Mock-провайдере. | Must |
